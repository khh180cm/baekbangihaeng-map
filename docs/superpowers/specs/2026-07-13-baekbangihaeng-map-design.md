# 백반기행 맛집 지도 — 설계 문서

- 작성일: 2026-07-13
- 상태: 승인됨 (설계 단계)
- 프로젝트 디렉토리: `penguin` (신규, greenfield)

## 1. 목적

허영만의 TV조선 예능 **《백반기행》**(2019~ , 다수 시즌/기수)에 등장한 식당들을,
**남한 지도에서 지역별로 눌러가며 빠르게 탐색**할 수 있는 정적 웹 페이지를 만든다.

각 식당에 대해 다음을 보여준다:
- 식당명, 대표 음식(카테고리), 메뉴 목록
- 주소
- 네이버 지도 / 카카오맵 바로가기 링크
- 대표 음식 사진(링크 연결 방식으로 제공)
- 출연 기수/회차 정보

## 2. 핵심 결정 사항 (확정)

| 항목 | 결정 | 근거 |
|------|------|------|
| 데이터 확보 | **전체 자동 수집 시도** (best-effort) | 공식 API/데이터셋 부재. 웹 리서치로 최대한 수집하고 신뢰도 플래그로 공백을 명시 |
| 지도 UI | **SVG 지도 + 식당 카드 리스트** | 가볍고 API 키 불필요, 정적 배포 가능 |
| 드릴다운 | **2단계**: 전국(시/도) → 시/도 확대(구/동/읍 구획 + 빨간 점) | 동 단위 지도는 비현실적; 동 정보는 카드 주소·필터로 표시 |
| 기술 스택 | **React + Vite 정적 SPA** (d3-geo) | 확장·유지보수 유리, 정적 호스팅 가능 |
| 이미지 | **링크 연결 방식** (카테고리 아이콘/플레이스홀더 + "사진 보기" → 네이버/카카오 플레이스) | 저작권·호스팅 문제 회피, 자동화 안정적 |
| 점 위치 정밀도 | **읍/면/동 중심점 근사** (키 불필요) | 경계 GeoJSON만으로 계산; 정확 위치는 외부 링크로 확인 |

## 3. 시스템 구조

두 개의 독립된 부분으로 나뉜다.

### (A) 데이터 파이프라인 — 오프라인/빌드 타임
웹 리서치 → 정규화·검증 → 좌표 부여 → `restaurants.json` 산출.
프론트엔드는 이 산출물만 소비한다(런타임 API 호출 없음).

### (B) 프론트엔드 — React + Vite SPA
`restaurants.json` + 경계 GeoJSON을 읽어 SVG 지도와 카드로 렌더링.
서버·API 키 없음, 정적 배포 가능.

## 5. 데이터 모델

식당 레코드 스키마:

```json
{
  "id": "seoul-jongno-그림집",
  "name": "그림집",
  "region": { "sido": "서울특별시", "sigungu": "종로구", "emd": "관철동" },
  "address": "서울 종로구 관철동 12-3",
  "coord": { "lat": 37.57, "lng": 126.98 },
  "menus": ["물회", "매운탕"],
  "signatureMenu": "물회",
  "category": "해산물",
  "episode": { "season": 12, "no": 145, "airDate": "2023-05-06" },
  "links": {
    "naver": "https://map.naver.com/v5/search/그림집 종로구",
    "kakao": "https://map.kakao.com/link/search/그림집 종로구"
  },
  "confidence": "high",
  "sources": ["namuwiki url", "blog url"]
}
```

필드 규칙:
- `links.naver`, `links.kakao`: 식당명 + 지역으로 **검색 URL 생성** (API 키 불필요).
- `coord`: 수집 단계가 아니라 **빌드 시** 주소의 읍/면/동 경계 중심점으로 계산. 같은 동에 여러 식당이 있으면 결정적(deterministic) 방식으로 살짝 흩뿌린다(jitter).
- `confidence`: `high` / `medium` / `low` — 자동 수집 신뢰도.
- `category`: 아이콘/필터용 유한 집합(예: 한식·해산물·고기·면·분식·기타).

## 6. 데이터 파이프라인 상세 (현실적 한계 포함)

`전체 자동 수집`의 현실을 명시한다.

**안정적으로 확보 가능**
- 식당명, 지역, 대표메뉴, 기수/회차: 나무위키 회차 목록·블로그 정리글에서 상당수 확보.
- 네이버/카카오 링크: 이름+지역으로 100% 자동 생성.

**불완전할 수 있음 → 명시적으로 처리**
- 정확한 지번/도로명 주소, 전체 메뉴/가격, 일부 오래된 회차.
- 주소 파싱 실패 시: 시군구 중심점에 배치 + "위치 미확인" 표시, `confidence: low`.
- 각 레코드에 `confidence`와 `sources`를 남겨 공백을 투명하게 드러낸다.

**수집 방식**
- 웹 검색 에이전트를 **Workflow로 팬아웃**해 시즌/지역 단위로 분담 수집.
- 서로 다른 출처를 **교차 검증**하는 단계를 거친다.
- 실행은 구현 단계에서 수행하고, 산출물을 검수한 뒤 프론트에 연결한다.

**v1 목표**: "완벽한 700곳"이 아니라 **"검증된 대다수 + 명확한 공백 표시"**.

## 7. 프론트엔드 컴포넌트

| 컴포넌트 | 역할 | 의존 |
|----------|------|------|
| `MapNational` | d3-geo로 17개 시/도 렌더, 클릭→시/도 선택, 지역별 식당 수 뱃지 | 시도 GeoJSON, 집계 데이터 |
| `MapProvince` | 선택 시/도의 시군구·읍면동 경계 path + 식당 빨간 점(동일 투영으로 배치), hover 툴팁 | 시군구/읍면동 GeoJSON, 해당 지역 식당 |
| `RestaurantPanel` | 현재 지역 식당 카드 리스트, 점과 양방향 하이라이트 | 필터 결과 |
| `RestaurantCard` | 상세: 카테고리 아이콘/플레이스홀더, 메뉴, 주소, 네이버/카카오 링크, "사진 보기", 기수/회차 | 단일 레코드 |
| `Filters` | 카테고리/기수/검색 UI + 필터 상태 | store |
| `store` | 선택 지역·선택 식당·필터 상태 (경량 상태관리) | — |
| `dataLoader` | 빌드된 `restaurants.json`·GeoJSON 로드(지역 단위 code-split 고려) | — |

**투영(projection)**: d3-geo를 사용. 전국 지도는 남한 전체 bounds에 fit, 시/도 확대 지도는 해당 시/도 bounds에 refit. 식당 점은 지도와 **동일한 투영**으로 변환해 정확히 얹는다.

**경계 데이터**: 공개 TopoJSON(예: `southkorea-maps`)을 시도/시군구/읍면동 레벨로 가공해 번들. 읍면동 중심점(centroid)은 좌표 계산에도 재사용.

## 8. 에러 처리 · 엣지 케이스

- 좌표 미확정 식당: "위치 미확인" 버킷으로 분리해 리스트에는 노출, 지도에는 시군구 중심 표시.
- 이미지/메뉴 누락: 플레이스홀더로 graceful 처리(레이아웃 깨지지 않음).
- 같은 동 다중 식당: 결정적 jitter로 점 겹침 완화.
- 데이터가 비어있는 시/도: 뱃지 0, 클릭 시 "등록된 식당 없음" 안내.

## 9. 테스트 전략

**데이터 파이프라인**
- 주소 파싱(시도/시군구/읍면동 추출) 단위 테스트
- 동 중심점 배치·jitter 결정성 테스트
- 스키마 검증: 필수 필드 존재, 좌표가 한국 bbox 내

**프론트엔드**
- 지도 렌더: 시/도 클릭 동작, 점 개수 == 해당 지역 데이터 수
- 필터 로직(카테고리/기수/검색) 컴포넌트 테스트
- Playwright 스모크: 전국 → 시/도 → 점 클릭 → 카드 표시 경로
